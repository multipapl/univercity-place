# Огляд поточної системи

Стан документа: описано те, що реально є в коді зараз, у поточній точці проєкту.
Це не план на майбутнє і не побажання, а спроба розкласти на людську мову, як цей viewer влаштований сьогодні.

## 1. Що це за додаток у своїй суті

Це не "рушій" і не універсальний редактор 3D-сцен.
По факту це спеціалізований браузерний переглядач однієї сцени, зібраний на `Three.js`, який:

- відкриває підготовлені `glb/gltf` сцени;
- ділить сцену на кілька окремих шарів;
- для різних типів об'єктів застосовує різні правила відображення;
- дає ходити/літати камерою;
- має локальні debug-інструменти для підкрутки вигляду;
- вміє підміняти окремі частини сцени живими ефектами, наприклад відео-вогнем.

Головна ідея тут така:
база сцени залишається "запеченою" з Blender, але поверх неї точково додаються окремі живі або керовані речі:
скло, відблиски, foliage, FX, фон, bloom, debug-корекції.

Тобто система вже не "просто показати один GLB".
Вона вже стала маленьким спеціалізованим 3D-додатком зі своїм пайплайном.

## 2. Загальна картина архітектури

На верхньому рівні все працює так:

1. `index.html` просто піднімає порожній контейнер `#app`.
2. `src/main.js` створює весь інтерфейс, renderer, сцену, камеру і зшиває всі модулі.
3. `src/config/viewerConfig.js` задає головні правила: які шари шукати, які параметри камери, які режими матеріалів, які дефолтні тюнінги.
4. `src/loaders/assetResolver.js` шукає, які саме файли реально доступні.
5. `src/loaders/sceneLayerLoader.js` завантажує знайдені шари сцени.
6. `src/materials/materialPipeline.js` для кожного mesh/material вирішує, як саме його треба показувати в браузері.
7. `src/materials/reflectionEnvironment.js` готує середовище для відбиттів.
8. `src/camera/navigationController.js` керує камерою, мишкою, клавішами і mobile-touch управлінням.
9. `src/diagnostics/performanceDiagnostics.js` рахує FPS, draw calls, трикутники і приблизну пам'ять текстур.
10. `src/debug/debugObjectInspector.js` дає локальний інспектор для вибору об'єкта і кольорових override'ів.

Якщо дуже коротко, то центр системи зараз один:
`main.js`.

Він не просто "точка входу".
Він одночасно:

- збирає весь UI;
- тримає глобальний runtime state;
- створює renderer і post-processing;
- запускає завантаження сцени;
- з'єднує між собою всі модулі;
- обробляє більшість подій інтерфейсу;
- запускає цикл кадрів `animate()`.

Тому головна складність керування проєктом зараз не в тому, що тут дуже багато файлів.
Складність у тому, що одна велика "центральна нервова система" сидить у `main.js`, а решта модулів підключені до неї як окремі вузли.

## 3. Як система запускається по кроках

Ось що реально відбувається після відкриття сторінки:

### 3.1. Старт сторінки

`main.js`:

- створює `viewport`;
- створює меню;
- створює loading screen;
- створює crosshair;
- створює mobile controls;
- додає все це в `#app`.

Тобто HTML майже порожній, а весь інтерфейс будується кодом на льоту.

### 3.2. Ініціалізація 3D

Далі `main.js` створює:

- `WebGLRenderer`;
- `Scene`;
- `PerspectiveCamera`;
- `Clock`;
- `PMREMGenerator` для reflection environment;
- окрему групу `sceneRoots`, куди кладуться завантажені сцени.

### 3.3. Читання режимів із URL і localStorage

На старті код читає:

- `?debug=1` для debug mode;
- `?assetBust=...` для примусового оновлення assets;
- `?lowMemoryBase=...` для економного режиму базових текстур;
- `?baseTextureCap=...` для обмеження максимального розміру базових текстур.

Також частина runtime-налаштувань читається з `localStorage`:

- `viewer.lowMemoryBaseMipmaps`
- `viewer.baseTextureMaxSize`

Тобто частина параметрів живе не лише в коді, а ще й у URL та браузерному локальному сховищі.

### 3.4. Підготовка завантажувачів

Створюються:

- `LoadingManager`
- `DRACOLoader`
- `GLTFLoader`
- `TextureLoader`

`LoadingManager` ще й автоматично додає cache-busting query до локальних asset URL, щоб браузер не показував старі версії.

### 3.5. Створення внутрішніх "станів"

У `main.js` є кілька великих plain-object станів:

- `uiState`
- `diagnosticsState`
- `backgroundState`
- `fireState`
- `reflectionState`

Вони не лежать в окремому state manager.
Це просто звичайні JS-об'єкти, які спільно змінюють різні частини програми.

### 3.6. Створення сервісів

Після цього `main.js` створює модулі:

- reflection environment manager;
- material pipeline;
- performance diagnostics;
- debug object inspector;
- navigation controller;
- scene layer loader.

Важливий момент:
ці модулі не живуть самі по собі.
`main.js` передає їм купу колбеків, станів і залежностей.

Тобто тут модель не "модулі самі один одного знаходять",
а "головний файл руками всіх знайомить між собою".

### 3.7. Перший запуск

Потім код:

- прив'язує обробники кнопок і слайдерів;
- синхронізує стартові UI-значення;
- викликає `sceneLayerLoader.loadSceneLayers()`;
- запускає нескінченний цикл `animate()`.

## 4. Система конфігурації: за що відповідає `viewerConfig`

Файл: `src/config/viewerConfig.js`

Це головне місце, де зібрані правила додатка.
Не вся логіка, але майже всі дефолтні значення і домовленості.

Тут лежать:

- базовий шлях до assets;
- список шарів сцени;
- параметри камери;
- режим руху;
- пресети матеріалів;
- runtime optimization;
- прості material tweaks;
- параметри post-processing.

### 4.1. Базовий шлях до assets

За замовчуванням сцена шукається в:

`/assets/scene`

Але це можна замінити змінною середовища:

`VITE_SCENE_ASSET_BASE_URL`

Ідея тут проста:
локально можна працювати з `public/assets/scene`,
а в staging/production підкласти зовнішнє сховище.

### 4.2. Шари сцени

Система очікує не одну сцену, а набір окремих шарів:

- `background`
- `bg360`
- `base`
- `alpha`
- `glass`
- `reflect`
- `fx`

З них обов'язковий тільки `base`.
Якщо `base` не знайдено, завантажується запасна тестова кімната-сітка.

У кожного шару є:

- свій `id`;
- зрозуміла назва для UI;
- назва query-параметра;
- тип матеріального режиму;
- список можливих назв файлу-кандидата.

Тобто система не жорстко прив'язана лише до одного імені файлу.
Вона вміє шукати кілька варіантів.

### 4.3. Камера і переміщення

Конфіг визначає:

- `fov`;
- висоту камери;
- subtle camera shake / ambient motion;
- режим `walk` або `fly`;
- стартову позицію;
- стартовий напрямок погляду;
- фіксовану висоту підлоги.

Зараз режим руху стоїть `walk`.

Це означає:

- горизонтальне переміщення дозволене;
- вертикальне ручне переміщення відключене;
- але це не справжня фізика ходьби.

Тут важливо:
поточний код не має системи колізій.
Камера не впирається в стіни і не перевіряє прохідність.
У режимі `walk` вона просто не літає вгору/вниз вручну.

### 4.4. Material presets

У конфігу багато пресетів для окремих матеріальних режимів:

- alpha cutoff;
- скло;
- FX;
- fire video;
- reflection UV-канали;
- reflection material defaults;
- background color/motion/warp parameters.

Тобто `viewerConfig` тут не лише "налаштування сцени".
Він ще і "словник правил відображення".

## 5. Система assets і пошук файлів

Файли: `src/loaders/assetResolver.js`, `src/config/viewerConfig.js`

Це окрема маленька підсистема.
Вона вирішує питання:
"де взяти сцену і які файли реально існують".

### 5.1. Як шукаються файли

Для кожного шару працює така логіка:

1. Якщо в URL явно передано посилання, воно має пріоритет.
2. Якщо ні, код перебирає список стандартних імен файлів.
3. Для кожного кандидата робиться `HEAD`-запит.
4. Береться перший файл, який реально відповів і не виглядає як HTML fallback.

Наприклад, `base` може бути:

- `scene.glb`
- `scene.gltf`

А `alpha` може бути:

- `alpha.glb`
- `leaves.glb`
- `leaf.glb`
- `foliage.glb`

### 5.2. Що це дає

Це дає гнучкість:

- можна міняти назви файлів без повного переписування коду;
- можна підставляти зовнішні URL через query params;
- можна мати кілька варіантів імен на перехідному етапі.

### 5.3. Що тут важливо пам'ятати

Оскільки пошук йде через `HEAD`, зовнішній storage або сервер має коректно відповідати на такі запити.
Інакше код може "не побачити" asset, навіть якщо файл фізично існує.

## 6. Система завантаження шарів сцени

Файл: `src/loaders/sceneLayerLoader.js`

Це модуль, який реально збирає сцену з кількох частин.

Його обов'язки:

- знайти доступні шари;
- завантажити кожен `glb/gltf`;
- пройтися по всіх mesh усередині;
- передати mesh у material pipeline;
- окремо підхопити fire video для FX;
- додати корені сцен у `sceneRoots`;
- зібрати список `loadedLayers`;
- викликати колбеки, які оновлять UI, performance, spawn і debug.

### 6.1. Як виглядає процес

Для кожного знайденого layer:

1. Оновлюється статус у loading/menu UI.
2. `GLTFLoader` завантажує файл.
3. Кореню шару ставиться `viewerLayerId`.
4. Кожному mesh усередині теж ставиться `viewerLayerId`.
5. Кожен mesh проходить через `convertMeshForLayer(...)`.
6. Якщо це `fx`, окремо пробується відео для вогню.
7. Якщо це `background`, root ще кладеться в `backgroundRoots`.
8. Layer додається до `sceneRoots`.

### 6.2. Якщо шар необов'язковий і не завантажився

Код не падає.
Він просто пише warning у console і йде далі.

### 6.3. Якщо не знайшли базову сцену

Тоді викликається `addFallbackScene()` з `main.js`.
Це проста wireframe-кімната і сітка, щоб додаток не був порожнім.

### 6.4. Fire video

У FX-шарі є окремий runtime-хак:

- код шукає `fire.mp4`, `fire.webm`, `fx.mp4` або `fx.webm`;
- створює `HTMLVideoElement`;
- робить із нього `VideoTexture`;
- знаходить матеріали, які за назвою схожі на вогонь;
- підміняє їм texture на відео;
- ще й проганяє через окремий shader patch для кольору і маски.

Тобто вогонь тут не просто частина GLB.
Це жива текстура, яка накладається поверх певних матеріалів під час виконання.

## 7. Material pipeline: головна "магія" відображення

Файл: `src/materials/materialPipeline.js`

Це, мабуть, найважливіший технічний модуль після `main.js`.
Саме тут вирішується, як об'єкти з Blender перетворюються на те, що реально рендериться в браузері.

Якщо коротко:
pipeline не довіряє сцені "як є".
Він багато чого перебудовує на льоту.

### 7.1. Що він робить загалом

Material pipeline:

- аналізує назви mesh/material;
- знаходить спеціальні tweak-и;
- нормалізує текстури;
- вибирає UV-канали;
- створює нові runtime-матеріали;
- інколи додає shader hooks;
- веде списки background/fire/reflect матеріалів для подальшого керування;
- дає debug-інспектору спосіб адресувати конкретний material target.

### 7.2. Режими матеріалів

У системі зараз є такі material mode:

- `baked`
- `background`
- `unlitAlpha`
- `alphaCutout`
- `glass`
- `reflect`
- `fx`

Кожен шар сцени під час завантаження прив'язаний до одного з цих режимів.

### 7.3. Що означає кожен режим

#### `baked`

Це базовий режим для основної сцени.
Зазвичай він переводить матеріал у дешевший для runtime вигляд, ближчий до "показати готову запечену картинку".

Суть:

- використовується простіший material;
- текстура вважається головним джерелом вигляду;
- для base-layer є окремі texture optimization.

#### `alphaCutout`

Це режим для foliage / leaves / cutout-геометрії.

Суть:

- включається alpha clip, а не м'яка прозорість;
- підтримується розділення color UV і alpha UV;
- є налаштування для mipmaps, anisotropy, premultiply alpha.

Тобто це окремий шлях саме для тонких вирізаних елементів.

#### `glass`

Скло тут не фізично точне, а стилізовано-прагматичне.

Суть:

- material робиться прозорим;
- додається shader patch з fresnel-подібною логікою;
- краї можуть бути виразнішими за центр;
- прозорість залежить від кута погляду.

Тобто ідея не "чесно симулювати скло",
а дешево дати відчуття скла.

#### `reflect`

Це окрема дорога для "геройських" відбивних поверхонь.

Суть:

- використовується `MeshPhysicalMaterial`;
- підтягуються roughness/metalness/ao/normal карти;
- призначається environment map;
- є окремі контролі для env intensity, IOR, specular, metalness.

Цей режим не про всю сцену.
Він про вибіркові об'єкти, яким потрібен кращий живий відгук на світло.

#### `fx`

Це режим для спеціальних ефектів.

Суть:

- матеріал прозорий;
- може бути additive blending для "вогняних" назв;
- може отримати відео як texture;
- саме цей шар потрапляє в selective bloom.

#### `background`

Це окремий custom `ShaderMaterial` для фонового шару.

Він не просто малює текстуру.
Він має:

- hue/saturation/value корекцію;
- warp рух UV;
- shimmer;
- можливість повільного руху в часі;
- окремий rotation root для фону.

Тобто background тут спеціально "оживлюється".

#### `unlitAlpha`

Це простіший шлях для прозорих/напівпрозорих "неосвітлених" речей, зокрема `bg360`.

Його сенс:
показати картинку максимально прямо, без складного освітлення.

### 7.4. Shader hooks

Одна з найважливіших причин складності цього проєкту:
pipeline не лише створює матеріали,
він ще і переписує частини shader-коду через `onBeforeCompile`.

Це використовується для:

- debug color correction;
- glass fresnel path;
- fire video patch;
- окремих material tweaks;
- розділених UV для alpha.

Тобто частина поведінки не лежить у "чистому" material API.
Вона вшивається в shader прямо під час компіляції матеріалу.

Саме тому цей код уже не схожий на "простий Three.js viewer".

### 7.5. Texture channel / UV логіка

Pipeline вміє:

- перевіряти, які UV-канали є в mesh;
- брати preferred UV;
- якщо його нема, падати назад на `uv`;
- насильно ставити texture.channel.

Це важливо для:

- foliage;
- reflect;
- різних карт roughness/metalness/ao.

Тобто частина домовленостей між Blender і viewer уже зав'язана на UV-канали.

### 7.6. Runtime texture optimization

Окремо pipeline вміє:

- вимикати mipmaps для base layer;
- зменшувати базові текстури до заданого розміру через canvas;
- перераховувати texture image на льоту.

Зараз ці оптимізації застосовуються саме до базового шару.

### 7.7. Material tweaks

У конфігу є простий механізм tweak-ів по назві mesh/material.
Зараз реально використовується tweak для background-подібних об'єктів.

Для чого це потрібно:

- трохи підкрутити яскравість/насиченість;
- виключити деякі mesh із розрахунку "ігрових" bounds.

Тут видно важливий принцип усієї системи:
частина логіки базується не на строгій схемі даних, а на домовленостях по назвах.

## 8. Reflection environment

Файл: `src/materials/reflectionEnvironment.js`

Це окремий сервіс для відбивних матеріалів.

Його задача:

- знайти картинку середовища для відбиттів;
- якщо вона є, підготувати її для runtime;
- якщо її нема, згенерувати запасний environment;
- покласти результат у `scene.environment`.

### 8.1. Звідки береться reflection map

Спочатку шукається URL через:

- query param `reflectEnv`
- або стандартні файли: `cubemap.png`, `cubemap.jpg`, `cubemap.jpeg`

### 8.2. Що відбувається далі

Якщо картинку знайдено:

- вона завантажується як equirectangular texture;
- проганяється через `PMREM`;
- готова texture стає environment map.

Якщо ні:

- створюється запасне "кімнатне" середовище через `RoomEnvironment`.

Це означає, що reflect-матеріали не залишаться зовсім "голими", навіть якщо власного cubemap нема.

## 9. Рендеринг і post-processing

Основна логіка сидить у `src/main.js`.

### 9.1. Базовий renderer

Renderer:

- antialias увімкнений;
- color space виставлений `SRGB`;
- shadow map вимкнені.

Тобто тут ставка не на динамічні тіні, а на baked-подачу плюс окремі runtime-ефекти.

### 9.2. Tone mapping і exposure

У меню можна міняти:

- tone mapping (`standard` / `none`);
- exposure.

Це звичайна глобальна корекція вже на рівні рендерингу.

### 9.3. Selective bloom

Є окремий bloom-конвеєр.
Але він не застосовується до всієї сцени.

Працює так:

1. FX-шару призначається спеціальний render layer.
2. Перед bloom-проходом усі "не-bloom" об'єкти тимчасово затемнюються в чорний.
3. Рендериться окремий bloom pass.
4. Потім bloom змішується з основною картинкою.

Сенс:
світитися мають лише вибрані FX-об'єкти, а не вся сцена.

### 9.4. Що зараз світиться

У поточному коді selective bloom прив'язаний саме до шару `fx`.
Не до окремих прапорів на material, а саме до layer id.

## 10. Камера, рух і керування

Файл: `src/camera/navigationController.js`

Це окремий модуль, який керує всім, що пов'язано з рухом.

Його обов'язки:

- pointer lock;
- mouse-look;
- клавіатура;
- touch joystick;
- touch look pad;
- boost/sprint;
- wheel для швидкості;
- стартова постановка камери;
- camera height;
- ambient motion.

### 10.1. Режими

Є два режими:

- `walk`
- `fly`

Зараз активний `walk`.

### 10.2. Що таке walk тут насправді

Це не персонаж з фізикою.
Немає:

- колізій зі стінами;
- сходинок;
- гравітації;
- raycast-ходіння по підлозі;
- системи "не можна пройти крізь меблі".

Є лише така домовленість:

- рух по X/Z дозволений;
- вертикальний ручний рух відключений;
- стартова висота камери розрахована як "підлога + eyeHeight".

Тобто це "режим пересування як у walkthrough", а не справжній character controller.

### 10.3. Pointer lock

На desktop:

- клік по viewport захоплює курсор;
- `Esc` відпускає;
- меню тимчасово знімає lock;
- після закриття меню курсор може захопитися назад.

### 10.4. Mobile

На touch-пристроях код показує:

- лівий joystick для руху;
- правий look pad для огляду;
- кнопки boost;
- у fly mode ще були б кнопки up/down.

Тобто mobile-керування тут уже не "заглушка", а реальна робоча схема.

### 10.5. Автоматичний старт камери

Позиція камери береться:

- або з `viewerConfig.locomotion.startPosition`;
- або з bounds сцени.

Для цього navigation controller обчислює розміри сцени.
Але важливий нюанс:
для цього розрахунку може ігноруватися фонова геометрія, якщо вона позначена tweak-ом як неігрова.

Це зроблено, щоб background dome або backdrop не ламали центр і розміри сцени.

### 11.2. Що є тільки в debug mode

У debug mode відкриваються додаткові секції:

## 12. Debug mode і локальний object inspector

Файли:

- `src/debug/debugObjectInspector.js`
- `src/debug/objectOverrideStore.js`
- `vite.config.js`
- `public/debug.scene-overrides.json`

Це одна з найцікавіших частин поточної системи.

### 12.1. Для чого він існує

Інспектор дає локально:

- вибрати об'єкт у сцені;
- побачити, з якого він шару;
- побачити назву mesh і material;
- підкрутити для цього target:
  - hue
  - saturation
  - value
  - gamma
- скопіювати JSON override'ів;
- у dev-режимі зберегти JSON у файл.

### 12.2. Як вибирається об'єкт

Працює raycast по viewport:

- натискаєш `Pick Object`;
- курсор у меню лишається активним;
- клікаєш по mesh;
- система знаходить material target;
- target описується не лише назвою, а й шляхом у сцені та номером material slot.

Тобто адресація тут доволі акуратна:
не просто "назва матеріалу", а більш стійкий ключ.

### 12.3. Де зберігаються override'и

Формат зберігання:

`public/debug.scene-overrides.json`

Структура проста:

- `version`
- `targets[]`

Кожен target містить:

- `layerId`
- `meshName`
- `meshPath`
- `materialName`
- `materialSlot`
- `hue`
- `saturation`
- `value`
- `gamma`

### 12.4. Як save працює технічно

У локальному dev server є спеціальний middleware у `vite.config.js`.
Він ловить `POST /__debug/scene-overrides` і записує JSON у `public/debug.scene-overrides.json`.

Важливо:
це працює тільки в локальному Vite serve-режимі.
У production-збірці збереження в файл немає.

### 12.5. Обмеження інспектора

Debug color correction не застосовується до всіх material типів.
Зокрема custom `ShaderMaterial` не підтримується цим механізмом.

Практично це означає:
деякі спецматеріали, наприклад background shader, не редагуються через цей інспектор так само легко, як звичайні mesh material.

## 13. Діагностика і performance-панель

Файл: `src/diagnostics/performanceDiagnostics.js`

Цей модуль збирає прості runtime-метрики.

Показує:

- FPS;
- час кадру;
- draw calls;
- triangles;
- кількість texture;
- приблизну texture memory.

### 13.1. Що тут важливо

Texture memory рахується приблизно, не абсолютно точно.
Це скоріше практичний орієнтир, ніж лабораторний вимір.

### 13.2. Що доступно лише в debug

Детальна статистика текстур і примітки про memory tuning показуються тільки в debug mode.

Там же можна:

- вимкнути mipmaps для базової сцени;
- обмежити максимальний розмір base texture.

## 14. Поточні домовленості по файлах і назвах

Це дуже важливий розділ, бо саме тут ховається багато "чому раптом щось не спрацювало".

### 14.1. Де лежать assets

Локально зараз основні assets лежать у:

`public/assets/scene/`

Там уже є:

- `scene.glb`
- `bg.glb`

### 14.2. Що з цього обов'язкове

Решта шарів необов'язкові.

Тобто це вже система з прихованими "контрактами по іменах".

## 15. Що і з чим взаємодіє

Нижче проста карта взаємодій людською мовою.

### 15.1. Хто головний

`main.js` керує всім.

Він:

- читає конфіг;
- створює UI;
- створює рендеринг;
- створює сервіси;
- викликає loader;
- оновлює UI по подіях;
- щокадру запускає рух, фонову анімацію, bloom і статистику.

### 15.2. Як зв'язаний loader

`sceneLayerLoader` залежить від:

- `viewerConfig`
- `assetResolver`
- `GLTFLoader`
- `materialPipeline`
- reflection environment
- статусних колбеків із `main.js`

Після завантаження він віддає назад у `main.js` список loaded layers.

### 15.3. Як зв'язаний material pipeline

`materialPipeline` залежить від:

- `viewerConfig`
- `backgroundState`
- `fireState`
- `reflectionState`
- reflection environment

Він не рендерить сцену сам.
Він лише перетворює mesh/material так, щоб сцена далі правильно жила в renderer.

### 15.4. Як зв'язаний navigation controller

`navigationController` залежить від:

- `camera`
- `renderer`
- `viewport`
- mobile UI elements
- `viewerConfig`

Він не знає, як завантажується сцена.
Але знає, як поставити камеру у стартову позицію після того, як сцена з'явилась.

### 15.5. Як зв'язаний debug inspector

`debugObjectInspector` залежить від:

- `camera`
- `sceneRoots`
- DOM canvas renderer'а
- `materialPipeline`
- debug UI елементів
- dev server endpoint для save

Тобто це локальний інструмент поверх уже живої сцени.

### 15.6. Як зв'язані performance diagnostics

`performanceDiagnostics` залежить від:

- `renderer.info`
- `diagnosticsState`
- списку loaded layers
- helper-функції для розміру текстур

Він нічого не керує.
Він тільки рахує і показує.

## 16. Де зараз ховається складність

Нижче не критика, а пояснення, чому цим уже важко керувати.

### 16.1. `main.js` занадто центральний

Файл уже виконує роль:

- bootstrap;
- UI builder;
- state container;
- event hub;
- render coordinator;
- integration layer;
- runtime control panel.

Тому будь-яка нова дрібна правка часто вимагає лізти саме туди.

### 16.2. Конфіг змішав кілька рівнів відповідальності

`viewerConfig` одночасно містить:

- шляхи до assets;
- параметри камери;
- material rules;
- performance defaults;
- постобробку;
- naming conventions.

Для невеликого проєкту це зручно.
Але коли система росте, стає важче зрозуміти, що є "художнім пресетом", що є "технічним контрактом", а що є "runtime-поведінкою".

### 16.3. Частина логіки сидить у назвах

Система використовує назви mesh/material/file як сигнали.
Це швидко і практично, але з часом починає плутати:

- де правило явне;
- де воно просто "домовленість";
- де достатньо перейменування, а де треба міняти код.

### 16.4. Частина поведінки сидить у shader patch'ах

Коли material змінюється через `onBeforeCompile`, поведінку вже важче знайти очима.
Вона не лежить в одному зрозумілому місці типу "ось готовий material preset".

### 16.5. Debug-інструменти вже корисні, але це ще не повноцінний авторинг-потік

Інспектор допомагає локально:

- вибрати target;
- спробувати HSV/gamma override;
- зберегти JSON.

Але це ще не замінює повністю зрозумілу систему "налаштування сцени".
Зараз це скоріше локальний технічний інструмент.

## 18. Що є "джерелом правди" в поточному стані

Якщо коротко:

- за правила сцени відповідає `viewerConfig`;
- за складання runtime-сцени відповідає `sceneLayerLoader`;
- за фінальний вигляд матеріалів відповідає `materialPipeline`;
- за рух і камеру відповідає `navigationController`;
- за інтеграцію всього разом відповідає `main.js`.

Але фактичне "остаточне джерело правди" по поведінці все одно зараз у коді, а не в одному окремому конфіг-файлі.

Чому:

- частина рішень сидить у конфігу;
- частина в умовах по назвах;
- частина в shader patch'ах;
- частина в `main.js`;
- частина в debug override JSON.

Тому, якщо щось дивно працює, причину часто треба шукати не в одному файлі, а в зв'язці кількох.

## 19. Короткий практичний висновок

Поточна система вже складається з кількох окремих рівнів:

- завантаження шарів;
- конвертація матеріалів;
- runtime-ефекти;
- керування камерою;
- debug/local authoring;
- діагностика;
- UI-керування всім цим.

Вона все ще відносно компактна за кількістю файлів, але вже не проста за логікою.

Найважливіше, що варто тримати в голові:

1. Це viewer зі спеціальним пайплайном під конкретну сцену, а не загальний рендерер.
2. Основна сцена складається з шарів, і кожен шар має свій material mode.
3. `materialPipeline` є серцем візуальної поведінки.
4. `main.js` є серцем інтеграції і тому саме він зараз найважче читається і підтримується.
5. Багато рішень уже зав'язані на назви файлів, mesh і material.
6. Debug mode вже є корисним локальним інструментом, але ще не є повною системою керування сценою.
