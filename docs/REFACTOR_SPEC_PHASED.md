# Специфікація на поетапний рефакторинг поточної системи

Стан документа: це практичне ТЗ на рефакторинг поточного коду без зміни загальної поведінки viewer там, де це окремо не обумовлено.

Документ написаний так, щоб його можна було дати в новий чат як робочу специфікацію.

## 1. Контекст

Поточний додаток вже виконує свою задачу, але код став важким для підтримки.
Основні проблеми:

- `src/main.js` перевантажений і змішує bootstrap, UI, runtime-стан, інтеграцію модулів і render loop.
- `src/config/viewerConfig.js` став занадто великим і містить різнорідні налаштування.
- `src/materials/materialPipeline.js` став монолітним і змішує кілька відповідальностей.
- `src/loaders/assetResolver.js` має зайву гнучкість для проєкту, де планується жорсткий naming contract.

Ціль не в тому, щоб “зробити красиво заради краси”.
Ціль у тому, щоб:

- спростити внесення дрібних правок;
- зменшити кількість місць, які треба тримати в голові;
- зробити структуру очевидною;
- прибрати зайву універсальність, яка не потрібна цьому проєкту;
- не зламати поточну сцену і поточний workflow.

## 2. Загальні правила для всього рефакторингу

### 2.1. Що потрібно зберегти

Після рефакторингу має зберегтися поточна поведінка:

- сцена запускається як і зараз;
- базовий `scene` layer завантажується;
- optional layers продовжують працювати;
- рух камери, pointer lock, mobile controls і debug mode не ламаються;
- fire video, background motion, reflect path і selective bloom не зникають;
- debug overrides продовжують працювати;
- поточний візуальний результат не повинен помітно змінитися без окремої причини.

### 2.2. Що не потрібно робити в цьому рефакторингу

Не потрібно:

- придумувати нову систему рендерингу;
- переписувати матеріали “з нуля” без потреби;
- вводити React, Vue або інший UI framework;
- робити загальний редактор сцени;
- робити повну data-driven систему на всі майбутні випадки;
- ламати поточний asset pipeline заради абстракцій.

### 2.3. Головний принцип

Рефакторинг має йти не в бік більшої універсальності, а в бік:

- більш жорстких контрактів;
- менших і зрозуміліших модулів;
- явних зв'язків;
- простішого коду.

## 3. Очікуваний результат після всіх етапів

Після завершення всіх кроків система має виглядати так:

- `main.js` є тонким інтеграційним файлом, а не місцем, де живе майже все;
- конфіги розкладені по доменах;
- asset loading працює по жорстких іменах файлів без складного перебору кандидатів;
- material pipeline стає диспетчером і тонким glue-layer, а не монолітом;
- кожен material mode має окрему реалізацію або окрему фабрику;
- UI зібраний в окремих модулях;
- runtime-логіка і DOM-логіка розділені;
- нову дрібну правку можна робити без постійного лазіння по 1000+ рядків.

## 4. Порядок виконання

Робити строго в такій послідовності:

1. Спрощення asset pipeline.
2. Розділення конфігів по доменах.
3. Винос UI з `main.js`.
4. Розрізання `materialPipeline`.

Причина такого порядку:

- asset contract треба спростити раніше, ніж далі будувати архітектуру навколо нього;
- конфіги краще розкласти до великого UI/material refactor;
- UI можна винести без ризику змінити render-логіку;
- `materialPipeline` варто різати останнім, бо це найризиковіший етап.

## 5. Етап 1. Спрощення asset pipeline

### 5.1. Мета

Прибрати зайву гнучкість у пошуку asset-файлів.
Viewer має працювати за жорстким naming contract, а не перебирати багато варіантів.

### 5.2. Що потрібно зробити

- Прибрати логіку перебору великого списку `candidates` для layer-файлів.
- Прибрати `HEAD`-пробу кількох імен файлів у `assetResolver`.
- Перейти до моделі “є один очікуваний файл для кожного layer”.
- Залишити можливість graceful fallback, якщо expected file не знайдено.
- Залишити можливість явного override через query params тільки якщо це реально потрібно для dev/debug workflow.
- Якщо override через query params залишається, він має бути простим і явним, без складного автопошуку.

### 5.3. Бажана форма після змін

Замість поточного підходу:

- `background` шукає кілька варіантів;
- `alpha` шукає кілька варіантів;
- `fx` шукає кілька варіантів;

має бути підхід:

- кожен layer має один офіційний filename;
- назви задаються в одному місці;
- loader або бере цей файл, або пише зрозумілий статус і пропускає optional layer;
- якщо не знайдено required base scene, вантажиться fallback scene.

### 5.4. Що саме змінити у коді

Переглянути і спростити:

- `src/loaders/assetResolver.js`
- `src/config/viewerConfig.js`
- за потреби `src/loaders/sceneLayerLoader.js`

### 5.5. Що має вийти в структурі

Asset config має явно описувати:

- `layer id`
- `label`
- `material mode`
- `required / optional`
- `expected filename`

Без списків із 4-6 альтернатив на один layer.

### 5.6. Критерії приймання

Етап вважається завершеним, якщо:

- код більше не перебирає велику кількість filename-кандидатів;
- `assetResolver` став коротшим і простішим;
- відсутній optional layer не валить app;
- відсутній required layer не крашить app, а веде до fallback state;
- naming contract описаний чітко й у коді видно, які імена є офіційними.

## 6. Етап 2. Розділення конфігів по доменах

### 6.1. Мета

Прибрати перевантаженість `viewerConfig` і зробити налаштування логічно згрупованими.

### 6.2. Проблема поточного стану

Зараз в одному файлі змішані:

- asset paths і layer contracts;
- camera settings;
- locomotion settings;
- post-processing;
- interface flags;
- runtime optimization;
- material presets;
- material tweaks.

Це робить файл незручним для читання і пошуку.

### 6.3. Що потрібно зробити

Розбити конфіг по змісту на кілька модулів.

Рекомендована структура:

```text
src/config/
  assetsConfig.js
  cameraConfig.js
  interfaceConfig.js
  diagnosticsConfig.js
  materialsConfig.js
  viewerConfig.js
```

Допускається інша схожа структура, якщо вона логічна.

### 6.4. Що куди винести

В `assetsConfig.js`:

- base asset url;
- layer contracts;
- expected file names;
- статусні тексти, якщо це зручно.

В `cameraConfig.js`:

- `camera`;
- `locomotion`;
- стартові положення;
- ambient motion.

В `interfaceConfig.js`:

- `showCrosshair`;
- те, що стосується UI-поведінки, а не рендеру.

В `materialsConfig.js`:

- `materialPresets`;
- `materialTweaks`.

В `diagnosticsConfig.js` або споріднений файл:

- runtime optimization defaults;
- debug/stat-related defaults, якщо вони реально потрібні окремо.

А `viewerConfig.js` має стати тонким агрегатором, який просто збирає ці частини разом.

### 6.5. Важливе уточнення

Не потрібно робити конфіг надто роздробленим.
Ціль не в тому, щоб мати 20 дрібних файлів.
Ціль у тому, щоб в одного файлу була одна понятна зона відповідальності.

### 6.6. Критерії приймання

Етап вважається завершеним, якщо:

- великий конфіг розкладений по доменах;
- у новій структурі легко знайти налаштування камери окремо від матеріалів;
- головний `viewerConfig` значно спростився;
- імпорти в інших модулях не стали хаотичними;
- runtime-поведінка після змін не зламалась.

## 7. Етап 3. Винос UI з `main.js`

### 7.1. Мета

Прибрати з `main.js` усе, що стосується побудови DOM, пошуку UI-елементів і значної частини UI-binding.

### 7.2. Проблема поточного стану

Зараз у `main.js` живе:

- HTML-розмітка HUD;
- loading screen;
- crosshair;
- mobile controls;
- пошук десятків DOM-вузлів;
- частина меню-логіки;
- частина debug-UI логіки;
- event binding для великої кількості елементів.

Через це `main.js` перестає бути інтеграційною точкою і стає "файлом про все".

### 7.3. Що потрібно зробити

Винести UI в окремі модулі.

Рекомендований поділ:

```text
src/ui/
  createViewerShell.js
  viewerDomRefs.js
  menuController.js
  debugPanelBindings.js
```

Допускається інша структура, якщо вона дає такий самий результат.

### 7.4. Мінімальні цілі етапу

Потрібно винести з `main.js`:

- створення всіх UI DOM-вузлів;
- HTML-рядки меню / loading / mobile controls;
- логіку збору DOM reference;
- базову логіку відкриття/закриття меню;
- UI event bindings, які не є частиною core render loop.

### 7.5. Що має залишитися в `main.js`

У `main.js` має залишитися:

- створення renderer, camera, scene;
- створення core services;
- з'єднання цих сервісів між собою;
- запуск `loadSceneLayers()`;
- render/update loop;
- мінімально необхідні integration callbacks.

Тобто `main.js` має бути coordinator-файлом, а не місцем, де руками збирається весь HUD.

### 7.6. Що важливо не зламати

Після виносу UI мають і далі працювати:

- toggle меню;
- debug mode toggle;
- sliders;
- layer visibility toggles;
- object inspector controls;
- performance section;
- loading screen;
- mobile controls.

### 7.7. Критерії приймання

Етап вважається завершеним, якщо:

- `main.js` помітно зменшився;
- у `main.js` більше не лежать великі HTML-блоки інтерфейсу;
- UI-модулі відповідають за DOM і взаємодію, а не core rendering;
- логіка меню і debug-панелі читається окремо від логіки рендеру;
- візуальна поведінка HUD не зламалась.

## 8. Етап 4. Розрізання `materialPipeline`

### 8.1. Мета

Зберегти поточну візуальну логіку, але прибрати монолітність одного великого файлу.

### 8.2. Проблема поточного стану

Поточний `materialPipeline` змішує:

- фабрики runtime-матеріалів;
- shader patches;
- texture normalization;
- UV fallback logic;
- debug color correction;
- target description for debug inspector;
- runtime texture optimization.

Через це він став складним для читання і зміни.

### 8.3. Що потрібно зробити

Рознести `materialPipeline` на кілька модулів без зламу поведінки.

Рекомендована структура:

```text
src/materials/
  materialPipeline.js
  textureUtils.js
  debugMaterialOverrides.js
  debugMaterialTargeting.js
  shaderPatches/
    applyViewerMaterialPatches.js
    applyGlassMaterialPatch.js
    applyFireVideoMaterialPatch.js
  factories/
    makeBakedMaterial.js
    makeAlphaCutoutMaterial.js
    makeGlassMaterial.js
    makeReflectMaterial.js
    makeBackgroundMaterial.js
    makeUnlitAlphaMaterial.js
    makeFxMaterial.js
```

Назви можна трохи змінити, якщо зберігається ясність.

### 8.4. Яка роль має лишитися у `materialPipeline.js`

Після рефакторингу `materialPipeline.js` має:

- створювати залежності для material-related модулів;
- виступати як dispatcher;
- знати, який `material mode` до якого factory прив'язаний;
- повертати назовні публічний API для решти системи.

Він не має лишатися місцем, де в одному файлі сидить усе.

### 8.5. Як мислити про структуру

Кожен material mode має мати свою окрему реалізацію.

Приклад логіки:

- `baked` має свій factory;
- `alphaCutout` має свій factory;
- `glass` має свій factory;
- `reflect` має свій factory;
- `fx` має свій factory;
- `background` має свій factory.

Окремо мають жити:

- texture helper-функції;
- shader patch helpers;
- debug target description;
- debug color override logic.

### 8.6. Важлива межа

Не треба намагатися винести “все в конфіг”.

У конфігу мають лежати:

- параметри;
- mapping-и;
- presets;
- naming-based rules, якщо вони ще лишаються.

А код реалізації shader/material factory має залишатися кодом, а не перетворюватися на псевдодекларативний конструктор.

### 8.7. Мінімальна функціональна вимога

Після рефакторингу мають зберегтися:

- baked path;
- foliage alpha cutout path;
- glass path;
- reflect path;
- background path;
- FX path;
- fire video patch;
- debug HSV/gamma overrides;
- texture optimization для base layer.

### 8.8. Критерії приймання

Етап вважається завершеним, якщо:

- `materialPipeline.js` став значно меншим;
- material factories рознесені по окремих модулях;
- shader patches рознесені окремо;
- debug override / targeting logic більше не захована серед фабрик матеріалів;
- поведінка viewer візуально не деградувала;
- код став простіше читати і знаходити потрібний material path.

## 9. Додаткове уточнення по navigation controller

Цей документ не вимагає окремого рефакторингу `navigationController` як одного з чотирьох основних кроків.
Але під час робіт потрібно перевірити, чи не варто зробити малий супутній cleanup:

- винести key bindings у конфіг або окрему map-структуру;
- відокремити input mapping від movement math;
- прибрати очевидні дублювання, якщо вони знайдуться.

Важливо:
це не має роздувати scope.
Якщо cleanup маленький і безпечний, його можна зробити.
Якщо він тягне окремий великий рефакторинг, його треба відкласти.

## 10. Додаткове уточнення по performance diagnostics

Поточна performance-панель не є джерелом абсолютної правди про GPU runtime.

Під час рефакторингу не потрібно переписувати її повністю.
Але бажано:

- не погіршити її читабельність;
- явно зберегти позначку, що texture memory є приблизною оцінкою;
- за можливості не маскувати короткі стартові просадки ще сильніше, ніж зараз.

Якщо в межах рефакторингу можна безпечно покращити frame-time reporting, це допустимо.
Але це не основна задача цього документа.

## 11. Технічні обмеження на виконання

### 11.1. Рефакторинг має бути поетапним

Кожен етап має бути завершуваним окремо.
Не треба робити “великий вибух”, де одночасно змінюється вся структура проєкту.

### 11.2. Після кожного етапу система має залишатися робочою

Після кожного етапу потрібно перевіряти:

- `npm run build`
- базовий запуск сцени
- роботу меню
- роботу debug mode
- відсутність очевидних runtime error у консолі

### 11.3. Не ламати поточні локальні debug-інструменти

Файл `public/debug.scene-overrides.json` і dev-save endpoint мають продовжувати працювати.

### 11.4. Не вносити непотрібну універсальність

Якщо стоїть вибір між:

- більшою гнучкістю;
- простішим жорстким рішенням під поточний пайплайн;

у межах цього проєкту треба обирати простіше жорстке рішення.

## 12. Формат здачі результату

Після виконання всіх етапів у новому чаті очікується:

- оновлена структура файлів;
- збережена працездатність viewer;
- короткий звіт по кожному етапу:
  - що змінено;
  - які файли з'явилися;
  - які старі відповідальності були винесені;
  - що спеціально залишено без змін;
- результати перевірки збірки;
- список можливих залишкових ризиків.

## 13. Коротке формулювання задачі для нового чату

Якщо потрібно коротко вставити завдання в новий чат, його можна сформулювати так:

Виконай поетапний рефакторинг поточного Three.js viewer без помітної зміни поведінки.
Цілі:

- спростити asset pipeline до жорсткого naming contract;
- розділити великий `viewerConfig` на доменні конфіги;
- винести весь UI і DOM-binding з `main.js`;
- розрізати монолітний `materialPipeline` на factories, shader patch modules, texture utils і debug-related modules.

Працюй строго по етапах.
Після кожного етапу система має залишатися робочою.
Не додавай зайву універсальність.
Оптимізуй під простоту підтримки, явні контракти і поточний жорсткий pipeline.
